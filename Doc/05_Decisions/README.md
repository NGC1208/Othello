# Decision Records

このフォルダーでは、ゲーム企画上の判断と技術上の判断を分けて記録する。

## 開発フロー

```text
Vision / Design Pillars
        ↓
DDR: プレイヤーへ何を、なぜ提供するか
        ↓
必要に応じてADR: DDRを技術的にどう実現するか
        ↓
Task: 今回の実装範囲と受け入れ条件
        ↓
Code / Tests
```

すべてのDDRにADRが必要なわけではない。複数の実装方式、保守性、性能、依存関係、互換性などの技術的トレードオフがある場合にADRを作成する。

## Design Decision Record（DDR）

保存先: `Design/`

DDRは企画側の判断を記録する。

- プレイヤーへ提供する体験
- ゲームルールやゲームモード
- 難易度やバランスの方向性
- UIに表示する情報
- 製品範囲と対象外
- 判断の背景と企画上のトレードオフ

DDRは実装手段を固定しない。技術方式が変わっても、提供する体験が同じならDDRは維持できる。

## Architecture Decision Record（ADR）

保存先: `Architecture/`

ADRはエンジニアリング側の判断を記録する。

- システム構造と責務分割
- アルゴリズムとデータ構造
- フレームワーク、ライブラリ、ツール
- 性能、テスト容易性、互換性
- 障害時の復旧方針
- 技術的な選択肢とトレードオフ

ADRは関連するDDRの意図を変更してはならない。DDRを満たせない場合は、実装上の妥協を独断で行わず企画判断へ戻す。
